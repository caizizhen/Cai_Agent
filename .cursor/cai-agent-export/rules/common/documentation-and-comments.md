## 文档与注释规则

> 适用范围：本仓库内的 README、架构文档、内联注释与 docstring。

### 1. README 与架构文档

- 顶层 `README.md` 需保持与实际实现一致：
  - 修改核心模块/CLI 子命令后，应评估是否需要同步更新 README；
  - 大型架构调整后，优先更新 `docs/ARCHITECTURE.zh-CN.md` 等架构说明。
- 文档以「对齐 Claude Code / Everything Claude Code」为核心主线，清晰标注：
  - 已实现能力；
  - 规划中的能力（Roadmap）；
  - 与参考项目的刻意差异。

### 2. 注释与 docstring

- 注释只记录**意图与约束**，避免「逐行翻译代码」：
  - 解释为什么要这样设计；
  - 标注边界条件与不显然的 trade-off。
- 公共函数/类建议提供简短 docstring，说明：
  - 功能；
  - 关键参数与返回值；
  - 可能抛出的异常或重要副作用。

### 3. 技能与规则文档

- `rules/` 与 `skills/` 中的文件应当：
  - 有清晰标题（技能/规则名称）与适用场景说明；
  - 使用分节结构（目标、步骤、注意事项等）；
  - 避免与现有文档重复，必要时可以引用其他规则/技能。

### 4. Agent 与文档的关系

- 在 Plan 阶段，若发现现有文档明显过时或缺失，应在计划中加入「补文档」步骤；
- 在执行阶段生成或修改文档时，应：
  - 避免泄露敏感信息（遵守安全规则）；
  - 保持术语与其它文档一致。
