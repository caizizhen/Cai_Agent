## 日志与错误处理通用规则

> 适用范围：本仓库内所有服务与工具；包括 CLI、Agent 内部逻辑与工具实现。

### 1. 错误分级与处理原则

- **可预期错误（业务/用户输入）**：
  - 优先通过返回值或受控异常向上抛出（例如自定义异常类型）；
  - 在靠近边界层（CLI/TUI/HTTP）将错误转为友好的用户提示。
- **不可预期错误（编程错误/系统异常）**：
  - 应记录完整堆栈（stack trace），但在终端输出中做适度裁剪；
  - 可以直接向上抛出，由统一异常处理逻辑收敛。

### 2. 日志内容与隐私

- **禁止直接输出敏感信息**：
  - 不在日志中打印完整 API Key、令牌、密码、私钥等；
  - 对此类字段进行打码（例如只显示前 4 位与长度）。
- **上下文信息**：
  - 记录操作对象（如工作区路径、文件路径）时应尽量使用相对路径；
  - 记录失败原因时包含简短的「做什么 + 结果」描述。

### 3. CLI / TUI 输出约定

- **用户可见错误**：
  - 一律输出到 `stderr`；
  - 文本需简洁，避免直接展示巨长堆栈；如必须展示，可提供 `--verbose` 开关或附加命令。
- **中间过程信息**：
  - 在 `--json` 等机器可读模式下，应避免混入非 JSON 文本日志；
  - 仅在非 JSON 模式下打印人类可读过程信息。

### 4. 重试与失败策略

- **带重试的操作**（如网络请求、工具调用）：
  - 务必对重试次数与退避时间做上限控制；
  - 发生多次失败后，应在上层汇总为明确的失败原因与最终状态，而不是静默吞掉。
- **Agent 工具调用失败**：
  - 工具实现应返回结构化文本（清晰起因、建议重试或放弃）；
  - 规划或执行时，遇到多次同类错误应主动提示用户可能的环境问题。

### 5. 约束执行方式

- 规划阶段：需要进行批量修改或新增模块时，Agent 应在计划中包含**错误处理与日志策略**一节；
- 执行阶段：生成代码时应主动加入合理的异常捕获与日志，且遵守本规则中的隐私与输出约束。
