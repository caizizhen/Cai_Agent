## 通用命名与目录结构规则

> 适用范围：本仓库内所有语言与项目；如无特别说明，Agent 在规划与实现时都应默认遵守本节规则。

### 1. 目录与模块划分

- **单一职责**：一个目录或模块聚焦一个清晰领域（如 `config/`、`llm/`、`tools/`），避免「杂物间」式模块。
- **分层清晰**：
  - 顶层：入口、配置、文档（如 `README.md`、`docs/`、CLI 入口等）；
  - 中层：领域逻辑（如 agent/workflow/graph 等）；
  - 底层：工具与适配层（如 http 客户端、文件工具）。
- **避免循环依赖**：禁止 A→B、B→A 双向依赖；如有需要，引入更底层的共享模块或接口抽象。

### 2. 命名风格（跨语言）

- **语义优先**：命名必须表达用途与作用域，避免 `data` / `info` / `stuff` 等模糊名。
- **布尔型**：使用 `is_` / `has_` / `should_` / `can_` 前缀（例如 `is_valid`, `should_retry`）。
- **集合与计数**：
  - 列表/集合：使用复数名，如 `files`, `sessions`, `rules`;
  - 计数：使用 `_count` / `_total` / `_num`，如 `tool_calls_count`。
- **避免缩写滥用**：非约定俗成的缩写不要使用；确需缩写时需在文档或注释中说明含义。

### 3. 函数与文件职责

- **函数长度**：除极简单工具函数外，函数体不宜超过 ~50 行；过长逻辑应拆分为子函数。
- **单一入口 / 单一出口优先**：
  - 入口：清晰参数与返回类型；
  - 出口：避免在多处随意 `return` / `raise` 导致路径复杂。
- **文件职责**：
  - 一个文件不应同时承担「业务逻辑 + CLI 解析 + 配置解析」等多种角色；
  - 如必须混合（例如 `__main__`），需通过清晰的内部函数边界划清层级。

### 4. Agent 与工具相关命名

- **工具名称**：
  - 工具函数以动宾形式命名，如 `read_file`、`search_text`、`run_command`；
  - 工具对外暴露的 `tool` 名也应与函数名保持一致或有简单映射规则。
- **状态结构**：
  - Agent 状态字段应清晰表达含义，例如：`messages`、`iteration`、`pending`、`finished`、`answer`；
  - 计数或统计字段统一以 `_count` / `_total` 结尾。

### 5. 约束执行方式

- **规划阶段**：在 `plan` 或其他只读规划模式中，Agent 应主动提醒不符合本规则的设计（例如模块职责过多、命名过于含糊）。
- **执行阶段**：
  - 生成/修改代码时优先遵守本规则；
  - 如需引用既有「不规范」命名（历史包袱），需在说明中指出，并尽量避免扩散。
