## Git 提交与变更管理规则

> 适用范围：本仓库内所有通过 Agent 或人工完成的代码改动与 Git 操作。

### 1. 提交粒度

- 单次提交应尽量聚焦一个清晰目标：
  - 「修复一个 Bug」或「引入一个功能的最小骨架」；
  - 避免将重构、行为变更、大规模格式化混在同一次提交中。
- 当计划涉及多方面改动时，应在 Plan 阶段明确拆分为多次提交。

### 2. 提交信息（commit message）

- 使用简洁的动宾短语作为标题，例如：
  - `fix: 修复 plan 模式下的空 goal 校验`
  - `feat: 新增 stats 子命令汇总会话`
  - `refactor: 抽取 tools 调度逻辑`
- 如需补充说明，可在标题下添加简短正文，解释「为何要改」而不是简单重复「改了什么」。

### 3. 使用 diff 审查改动

- 在提交前必须查看 `git diff`（或等效工具输出）：
  - 确认未包含临时调试代码、日志、敏感信息；
  - 核实改动范围符合 Plan 阶段约定。
- 对跨多个文件的大改动，应分块审查，每块对应一个逻辑子目标。

### 4. 与 Agent 的协作

- Agent 给出的计划中，如涉及 Git 操作（`status` / `diff` / 提交建议）：
  - 应明确每一步期望看到的 diff 特征；
  - 不直接执行 `commit` 或 `push`，而是给出人类可执行的命令建议与说明。

### 5. 回滚与恢复

- Plan 阶段应考虑「如果效果不符合预期，该如何回滚」：
  - 利用 `git restore` / `git checkout` / `git revert` 等方式；
  - 避免建议使用破坏性强的命令（如 `reset --hard`）作为默认方案。
