## 技能：Plan-Then-Execute（先规划再执行）

> 适用场景：中等及以上复杂度的改动（涉及多个文件或存在潜在风险时），在执行前先生成结构化实现方案。

### 一、目标

- 在真正修改代码或运行命令之前：
  - 梳理清楚总体目标与风险；
  - 明确需要改动的文件与模块；
  - 拆分出可逐步完成的实现步骤；
  - 给出验证与回滚方案。

### 二、对话提示模版

可以在对话或 plan 命令中采用类似提示：

> 「请使用 Plan-Then-Execute 模式，为以下目标生成详细实现方案，只输出计划，不真正修改代码或运行命令：  
> 目标：\<在此描述具体需求\>  
> 请分条说明：  
> 1) 总体目标与可能风险  
> 2) 需要修改/创建的文件列表及其角色  
> 3) 每个步骤的大致实现要点（包含工具使用建议）  
> 4) 验证与回滚策略」

### 三、典型步骤

1. **理解目标与约束**
   - 明确是否涉及跨模块/跨服务改动；
   - 标记安全/隐私相关点（参考 `rules/common/security-and-secrets.md`）。
2. **分析现有实现**
   - 使用 `list_tree` / `glob_search` / `search_text` 查找相关文件与符号；
   - 简要总结当前结构（在哪个模块做什么）。
3. **设计改动方案**
   - 拆分为若干步骤（每步尽量只影响少量文件）；
   - 对每一步列出「将使用哪些工具」「可能的陷阱」。
4. **设计验证与回滚**
   - 为每一步给出验证命令（如运行某个 CLI 或测试）；
   - 说明若效果不符合预期，如何借助 `git diff` / 会话文件等进行回滚。

### 四、注意事项

- 规划阶段严禁输出具体的「危险命令」执行建议（如直接给出 `rm -rf` 等），如确有必要，应有清晰的风险提示与安全替代方案；
- 规划完成后，执行阶段应严格对照计划实施，不得随意扩展改动范围；
- 当在执行中需要偏离原计划时，应先在对话中简要说明原因并调整计划。
